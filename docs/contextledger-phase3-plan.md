# ContextLedger Phase 3 — Onboarding, Visual Editor & Integrations

**For Claude Code. Complete Phase 2 first.**

---

## What Phase 3 Adds

Phase 1: Core infrastructure (protocols, backends, CMV, three-tier memory, skill profiles, CLI)
Phase 2: Multi-skill project manifests, auto-routing, cross-skill fusion
Phase 3: Accessible onboarding, visual tooling, ecosystem integrations

The goal of Phase 3 is reducing adoption friction.
Currently ContextLedger requires you to write YAML, understand DAG structure,
and be comfortable with CLI. Phase 3 removes those requirements.

## Feature 1: Skill Extractor — Import Existing Workflows

**The problem it solves:**
Most potential users already have workflows. They have Python pipelines,
LangChain chains, extraction functions, Claude Code skills.
Asking them to write a profile.yaml from scratch is high friction.
The extractor generates a starter profile from existing code or examples.

### 1A: Python Function Extractor

Reads a Python file, identifies functions that look like extraction/synthesis logic,
and generates a profile stub.

```
ctx extract --from my_pipeline.py --output profile.yaml
```

**How it works:**

```python
# skill/extractor.py

class PythonExtractor:
    """
    Generates a profile.yaml stub from an existing Python file.
    Looks for:
    - Functions with 'extract', 'parse', 'analyse', 'synthesise' in the name
    - Functions that return dicts with 'findings', 'entities', 'results' keys
    - Docstrings that describe what the function extracts
    """

    def extract(self, file_path: str) -> ProfileYAML:
        source = open(file_path).read()
        tree = ast.parse(source)

        # Find candidate functions
        candidates = self._find_candidates(tree)

        # Map to DAG node types
        nodes = []
        for fn in candidates:
            node_type = self._infer_node_type(fn)
            nodes.append(DAGNode(
                id=fn.name,
                type=node_type,
                depends_on=self._infer_dependencies(fn, candidates),
            ))

        # Generate entity types from return values and docstrings
        entities = self._infer_entities(candidates)

        return self._build_profile_stub(nodes, entities, file_path)

    def _infer_node_type(self, fn) -> str:
        name_lower = fn.name.lower()
        if any(kw in name_lower for kw in ["extract", "parse", "read", "load"]):
            return "extraction"
        elif any(kw in name_lower for kw in ["reason", "analyse", "analyze", "relate"]):
            return "reasoning"
        elif any(kw in name_lower for kw in ["synthesise", "synthesize", "summarise", "summarize", "report"]):
            return "synthesis"
        elif any(kw in name_lower for kw in ["filter", "clean", "validate"]):
            return "filter"
        return "extraction"  # default
```

### 1B: Claude Code Skill Importer

Reads an existing `.claude/skills/SKILL.md` file and generates a profile.yaml
that wraps it.

```
ctx import --from .claude/skills/frontend-design/SKILL.md --output profile.yaml
```

**How it works:**

```python
class ClaudeSkillImporter:
    """
    Wraps an existing SKILL.md in a ContextLedger profile.
    The SKILL.md becomes a reference document in refs/.
    The profile YAML describes the skill's intent for extraction purposes.
    """

    def __init__(self, llm_client):
        self.llm = llm_client

    def import_skill(self, skill_md_path: str) -> ProfileYAML:
        content = open(skill_md_path).read()

        # Use LLM to infer entities and domain from SKILL.md content
        prompt = f"""
        This is a Claude Code skill definition:

        {content[:2000]}

        Infer:
        1. What domain is this skill for? (e.g. frontend development, security testing)
        2. What entities would be extracted from sessions using this skill?
           (e.g. components, vulnerabilities, findings, decisions)
        3. What data sources does it work with?

        Return JSON only: {{"domain": "...", "entities": [...], "sources": [...]}}
        """

        inferred = self.llm.complete_structured(prompt)
        data = json.loads(inferred)

        return ProfileYAML(
            name=Path(skill_md_path).parent.name + "-skill",
            version="1.0.0",
            extraction=ExtractionConfig(
                entities=data["entities"],
                sources=data["sources"],
            ),
            refs=[skill_md_path],  # SKILL.md becomes a reference doc
            # Standard DAG
            synthesis=SynthesisConfig(dag=DEFAULT_RESEARCH_DAG),
            session_context=SessionContextConfig(
                mode="skill_versioning",
                cmv_enabled=True,
            ),
        )
```

### 1C: Example-Based Skill Creator

User provides 3–5 examples of findings they want to extract.
ContextLedger infers the extraction rules.

```
ctx new my-skill --from-examples examples.json
```

Where `examples.json` is:

```json
[
  {
    "session_excerpt": "The users table has no index on email column.",
    "finding": {"type": "missing_index", "table": "users", "column": "email"}
  },
  {
    "session_excerpt": "Query on orders.created_at takes 4.2 seconds.",
    "finding": {"type": "slow_query", "table": "orders", "column": "created_at"}
  }
]
```

The extractor uses few-shot prompting to infer extraction rules from examples,
then generates the `extraction.rules` section of the profile.

## Feature 2: Visual Skill Editor

**The problem it solves:**
DAG structure and YAML are not intuitive for first-time users.
A browser UI lowers the barrier significantly.

**Scope:** Lightweight local web server. Not a hosted service.
Run `ctx editor` and it opens in your browser at `localhost:7432`.
No cloud dependency.

### Architecture

```
contextledger/
└── editor/
    ├── server.py          # FastAPI local server
    ├── api/
    │   ├── profiles.py    # CRUD for skill profiles
    │   ├── dag.py         # DAG visualization endpoints
    │   └── diff.py        # Profile diff endpoint
    └── ui/                # Single-file React app (no build step)
        └── index.html     # Self-contained, loads from CDN
```

**FastAPI endpoints:**

```python
# editor/server.py

GET  /api/profiles              # list all profiles
GET  /api/profiles/{name}       # get profile
POST /api/profiles/{name}       # save profile
GET  /api/profiles/{name}/dag   # get DAG as nodes+edges for visualization
GET  /api/diff/{a}/{b}          # section-by-section diff
POST /api/merge/{fork}/{parent} # trigger merge with tier result
GET  /api/status                # registry stats
```

**CLI command:**

```bash
ctx editor [--port 7432] [--no-browser]
```

Opens browser automatically. `Ctrl+C` to stop.

### UI Features

**Tab 1 — Profiles**
- List all skill profiles with version, parent, last modified
- Click to open profile editor
- Fork button → creates fork immediately
- Compare button → opens diff view

**Tab 2 — DAG Editor**
- Visual DAG with drag-and-drop nodes
- Node types shown as colored boxes (extraction=blue, reasoning=yellow, synthesis=green, filter=grey)
- Edges show dependencies
- Click node → edit node config inline
- Add node → dropdown of node types
- Delete node → removes node and validates no dangling dependencies

**Tab 3 — Diff View**
- Side-by-side section comparison
- Sections highlighted: changed (yellow), added (green), removed (red)
- Merge button per section (applies Tier 1 auto-merge for non-conflicting sections)
- Shows Tier 2 evaluation results inline when available

**Tab 4 — Context Browser**
- Query box at top
- Results from three-tier memory with tier indicators
- Filter by profile, date range, tier
- Click finding → expand to full session context

### Implementation Notes

- Use React loaded from CDN (no npm build step required)
- Use Cytoscape.js for DAG visualization (CDN)
- The UI is a single `index.html` file served by FastAPI
- All state is in ContextLedger's existing backends — editor has no own state
- The editor is read-write but all writes go through existing `ProfileParser` and `RegistryBackend`

## Feature 3: VS Code Extension

**The problem it solves:**
Switching to a terminal to run `ctx query` breaks flow.
The extension puts ContextLedger inside the editor where you already are.

**Scope:** Minimal VS Code extension. Not a full IDE integration.

### Extension Features

**Sidebar panel — ContextLedger**
- Shows active skill profile
- Recent findings (last 10)
- Quick query box

**Command palette commands:**
- `ContextLedger: Query` → opens input box, shows results in output panel
- `ContextLedger: Switch Profile` → quick-pick from available profiles
- `ContextLedger: Ingest Current Session` → manually triggers ingestion of current chat
- `ContextLedger: Show Status` → opens status in output panel

**Auto-routing integration:**
When you open a file, the extension calls `ctx project route --file <current_file>`
and shows the active skill in the status bar.

**Status bar item:**
`⟨ sdk-skill ⟩` — shows active skill, click to switch

### Implementation

```
vscode-contextledger/
├── package.json
├── src/
│   ├── extension.ts       # activation, command registration
│   ├── sidebar.ts         # sidebar tree view
│   ├── statusBar.ts       # active skill indicator
│   └── ctxClient.ts       # calls ctx CLI via child_process
└── README.md
```

`ctxClient.ts` just shells out to the `ctx` CLI — no reimplementation of logic.

```typescript
// ctxClient.ts
import { exec } from 'child_process';

export async function query(text: string, profile?: string): Promise<string> {
    const profileFlag = profile ? `--profile ${profile}` : '';
    return new Promise((resolve, reject) => {
        exec(`ctx query "${text}" ${profileFlag}`, (err, stdout) => {
            if (err) reject(err);
            else resolve(stdout);
        });
    });
}

export async function status(): Promise<object> {
    return new Promise((resolve, reject) => {
        exec('ctx status --json', (err, stdout) => {
            if (err) reject(err);
            else resolve(JSON.parse(stdout));
        });
    });
}
```

## Feature 4: GitHub Actions Integration

**The problem it solves:**
Teams using ContextLedger with a shared skill registry need automated validation
when skill profiles change. PRs that modify `profile.yaml` should trigger
a semantic diff and post results as a PR comment.

### Two Actions

**Action 1: `contextledger/skill-diff-action`**

Runs on PRs that modify any `profile.yaml`:

```yaml
# .github/workflows/skill-diff.yml
on:
  pull_request:
    paths:
      - '**/profile.yaml'

jobs:
  skill-diff:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: contextledger/skill-diff-action@v1
        with:
          changed-profiles: auto          # detects changed profiles automatically
          post-comment: true              # posts diff as PR comment
          anthropic-api-key: ${{ secrets.ANTHROPIC_API_KEY }}
```

The action:
1. Detects which `profile.yaml` files changed
2. Runs `ctx diff` on each changed profile (current vs base branch)
3. Posts a structured comment showing section-by-section diff
4. If Tier 2 evaluation data exists, includes precision/recall/novelty scores
5. Fails the check if Tier 3 conflicts are present (blocks merge until resolved)

**Action 2: `contextledger/skill-validate-action`**

Validates that all declared profiles in `project.yaml` exist in the registry:

```yaml
- uses: contextledger/skill-validate-action@v1
  with:
    project-manifest: .contextledger/project.yaml
```

## Feature 5: LangChain Integration

**The problem it solves:**
Many teams already use LangChain. They shouldn't need to rewrite anything
to get ContextLedger versioning on top of their chains.

### LangChain Callback Handler

```python
# integrations/langchain_handler.py

from langchain.callbacks.base import BaseCallbackHandler
from contextledger.mcp.server import ContextLedgerMCP

class ContextLedgerCallbackHandler(BaseCallbackHandler):
    """
    LangChain callback that automatically ingests chain runs
    into ContextLedger's three-tier memory.

    Usage:
        from contextledger.integrations.langchain_handler import ContextLedgerCallbackHandler

        handler = ContextLedgerCallbackHandler(profile="my-chain-skill")
        chain.invoke(input, config={"callbacks": [handler]})
    """

    def __init__(self, profile: Optional[str] = None):
        self.server = ContextLedgerMCP()
        self.profile = profile
        self._current_session = []

    def on_llm_start(self, serialized, prompts, **kwargs):
        self._current_session.append({
            "role": "user",
            "content": prompts[0] if prompts else "",
        })

    def on_llm_end(self, response, **kwargs):
        text = response.generations[0][0].text if response.generations else ""
        self._current_session.append({
            "role": "assistant",
            "content": text,
        })

    def on_chain_end(self, outputs, **kwargs):
        if self._current_session:
            self.server.ctx_ingest({
                "session_id": str(uuid.uuid4()),
                "messages": self._current_session,
                "profile": self.profile,
            })
            self._current_session = []
```

## Marketing Positioning

**The honest question:** Do you need a marketing doc? And where does it go?

The answer depends on what you're doing with ContextLedger:

**If you're shipping it as an open-source tool (GitHub):**
The README is your marketing. The first 10 lines of the README are what developers see when deciding whether to star or ignore it. You don't need a separate marketing doc. You need the README to lead with the right problem statement and the right differentiator. What to put at the very top:

```markdown
# ContextLedger

You iterate AI workflows across multiple domains.
Each domain needs different extraction rules.
You want to merge improvements back without losing reproducibility.
No existing tool handles this.

ContextLedger does.

**Skill versioning:** Fork a workflow for a new domain. Iterate independently.
Merge back with semantic conflict resolution that scores whether your changes
actually improve findings — not just whether the text differs.

**Second brain:** Unified context across Claude Code, Chat, Cursor, OpenAI.
Query across all your sessions from anywhere.

pip install contextledger
```

**If you're pitching it to people/investors:**
Then yes, a separate positioning doc makes sense. It would cover:
- Target audience (security researchers, ML engineers, research teams)
- The gap in the market (nobody combines skill versioning + semantic merge)
- Differentiators vs closest competitors (Mem0, PromptLayer, Spring AI)
- Use cases with concrete examples (Agent Prober, GenOpt, Veritas)
- Adoption path (open source → team tier → enterprise)

Want me to write both: the tightened README opening AND the separate positioning doc?
That way you have the README fixed for GitHub, and the positioning doc for
whenever you want to pitch it.
